Xogito: The Architecture of Auditable Reasoning

Design decisions, alternatives, and trade-offs.

Part I · The problem

1Reasoning Fails Through Bookkeeping

Ask a top AI model a hard question, and it will often reason well at every step: the math is right, the facts sound true, each piece of logic holds up on its own. But the final answer can still be wrong — stated with full confidence — and nothing in the output tells you which starting assumptions you should actually trust.

The failure follows a clear pattern: the model loses track of what it truly knows versus what it's just guessing. An early guess gets treated as fact later on. When two claims disagree, one just quietly disappears. A guess gets used to back up another guess. These are record-keeping failures, not thinking failures. Xogito is built on one idea: move that record-keeping outside the model, make it explicit, and enforce it with code. Do that, and a normal LLM is already good enough to reason well.

The result is a set of conclusions you can check. Every claim carries a confidence label it had to earn step by step, backed by evidence you can trace, from a process that's fully logged. The real question isn't "how do we make the model smarter?" It's "what design makes the record-keeping impossible to break quietly?"

2The First Commitment: LLM Proposes, Code Disposes

Should the model's output be allowed to change the system's state directly? Most agent frameworks say yes, then add checks afterward. Here, a bad write means a corrupted belief — and a corrupted belief means the product has failed. And when agents pass loose text to each other through shared memory, mistakes get copied so many times you can no longer tell where they started.

Xogito uses one rule from secure system design everywhere: whoever proposes an action never gets to approve it. Model output is always a proposal, and every proposal must pass a fixed code check before it can touch the system's state. The model does all the judging and has no power to act; the code has all the power to act and does no judging.

Three rules follow from this:

Make bad states impossible to write.
Where possible, don't just check that a rule was followed — build the system so breaking it can't even be expressed. A dependency loop that can't be written beats a checker that looks for loops.
Use labels, never numbers.
Model judgments use small, fixed sets of words (fits / partially fits / doesn't fit; strong / adequate / weak), never scores. A model-made "7/10" changes if you reword the question, doesn't repeat reliably, and invites math on a judgment that was never that precise.
Limits live in code.
Every cap, threshold, and budget sits in code (parametres.py) and never appears in a prompt. Tell a model "at most ten tasks" and it pads the list to ten; give it an invisible limit instead, and it proposes only what the problem actually needs. Graders never see the generators' scoring rules either — the standing defense against a system gaming its own metrics.

These rules shape the whole pipeline: a vague question first becomes an approved, clear specification (§3); rounds of planning, doing, and checking then build up claims backed by evidence (§6 through §10); and the run ends with a report that shows its own reasoning (§12).

Part II · The machine

3From Words to Problem: Intake and Formalization

Formalization is underdetermined

A request like "help me decide whether to launch this product in Algeria" first has to become a clear goal, a set of limits, and a way to measure success — before any planning can start. The simplest fix, one LLM call that says "restate this as a formal problem," fails on exactly the cases that matter, because there's no single right way to turn a loose question into a formal one. The same goal could be framed as a search for the best option, a search for any option that clears some bar, a game between multiple parties, or an ongoing feedback loop — and each framing sets different goals and produces a different kind of answer.

The philosopher Carnap called this kind of choice explication: none of the candidate framings is simply right or wrong — some are just better than others, judged by how closely they match the original idea, how precise they are, how useful they turn out to be, and how simple they are. A single LLM call would quietly pick one framing without weighing the rest — usually the "find the best option" framing, since that's what shows up most in its training data — and the user would never learn that a debatable choice was made for them. The biggest source of ambiguity in the whole pipeline would get settled by accident.

The five-phase pipeline

Phase 1: Characterize. One LLM call pulls out basic facts about the problem: does success come in degrees, is there one actor or several, is this a one-time decision or an ongoing process. It's kept blind to the list of possible framings, because a model that already knows "find the best option" is on the menu will shape its answers to justify picking it. The outputs are fixed categories, checked by code.

Phase 2: Filter. No LLM here at all. A fixed list, owned by code, says which framings fit which facts, and filtering is just matching one set against another. The lesson: once a question can be turned into structured data, hand it to code instead of the model — code always answers the same question the same way.

The catalog of framings lives in its own file, grown from live-run evidence to twenty structures — decisions, diagnoses, claim checks, forecasts, estimates, designs, schedules, rankings, and more — each with a plain-language definition and one concrete example query, because a classifier matches real problems against real problems better than against theory. What made the non-decision shapes reachable was one extra extracted characteristic: what the asker actually wants handed back — a decision, an explanation, a forecast, an artifact, a verdict on stated claims, a ranking. The first six characteristics couldn't tell "pick an option" apart from "explain a failure," so everything used to get funneled toward the decision-shaped framings.

Phase 3: Classify and select. One call labels each surviving framing as fits, partially fits, or doesn't fit, and is told plainly that labeling everything "doesn't fit" is a fine outcome. Code then groups the results by label and trims the list to a limit the model never saw. If the cutoff falls between two framings with the same label, that gets logged as boundary tension — an honest note that the cutoff was a bit arbitrary.

Phase 4: Generate. Each chosen framing gets written up in full. The model is told to treat each framing on its own and to note what that framing leaves out or changes.

Phase 5: Evaluate and decide. One last call rates each candidate against Carnap's four criteria as strong, adequate, or weak, but isn't allowed to pick an overall winner — code counts the labels and decides. If the top two candidates tie, the outcome is marked contested and the record shows selected_structure_id: None. The system never breaks a tie on its own.

Contested records and ratification

The only approved way to read the result, a function called get_final_formalization, returns nothing when the record is contested: nothing further down the line can move forward on a tie, because there's no sensible way to proceed on one. A human steps in instead: the user sees the tied framings and what each one means in practice ("find the single best option" versus "find any option that clears these bars") and picks.

Tied or not, the finished specification then goes to the user for ratification: they confirm the goal, the limits, how success will be measured, and the assumptions behind all of it. A flag, ratified: true, gets checked by a function called assert_spec_ratified at the start of every step that follows. That check is repeated on purpose everywhere downstream, since missing it in even one place would fail silently.

One more thing gets extracted before ratification: a checklist of every sub-question the user explicitly named ("verify both of Dana's claims" is two entries plus the decision itself). It's ratified along with the spec, and it carries real weight later — the run isn't allowed to call itself finished while any named item still lacks a completed task covering it (§12). This exists because a live run once answered the main question well and silently skipped a named sub-question; a question the user asked by name is never optional.

Contextual anchors

Facts the user states as fixed ("the budget is $40k") are called anchors. Code, not an LLM, pulls these out of the raw text using clear markers, since an LLM might reword them without meaning to. They're then stored word-for-word and dropped word-for-word into every prompt that needs them, always labeled as coming from the user. Rewording would count as reinterpreting, which is exactly the kind of quiet drift the system is built to stop. If later evidence conflicts with an anchor, the system raises a flag instead of overriding what the user said.

4Memory That Cannot Lie: The Workspace

Where should this state actually live? The usual multi-agent answer, letting each agent keep its own state and merging it later, doesn't work here, because conclusions depend on each other. When two executors disagree about the same fact, something needs to notice; the final report needs one version to point to as authoritative; and a challenged conclusion needs to be traceable back to where it came from. Scattered state with no single source of truth ends up holding beliefs nobody actually decided to hold.

The workspace is a single store that holds everything a run currently believes. It can only be changed through a narrow set of guarded methods, and every change is logged. Think of it like a shared whiteboard that specialists read and write to — with one added rule: agents don't write to it directly. They submit a proposed change, and the workspace's own code decides what actually gets written.

Entities

The workspace stores three kinds of item, each a simple data structure:

  • Task: an assignment. Description, kind, dependencies, lifecycle status, and a checkable completion contract (done_when, supplied by the planner, §6).
  • Claim: a tracked assertion. Statement, belief label, and the list of artifacts serving as evidence.
  • Artifact: a work product. Full content, a one-line summary, and the producing task.

A claim and the artifact that backs it are linked both ways: the claim lists its evidence, and the artifact lists the claims it supports. Both lists are updated together by one pair of functions, link_evidence and unlink_evidence, so they can never fall out of sync. A more standard database design would use a separate table to hold extra details about each link; that's been set aside for now, until links actually need those extra details, which is expected to happen soon in §9.

Access rules

Every method that reads data hands back a full copy, so outside code can never hold a live connection to it. Changing that copy does nothing to the real data. Methods that write data work on the real storage directly instead — reading through a copying method and then editing the result would just edit and throw away a copy, a trap worth naming. Every write also checks that its references are valid and logs itself automatically.

Provenance is the permanent, ever-growing record of every change: who did what, to which item, and when. "Where did this belief come from?" becomes something you can simply look up. Nothing is ever deleted: rejected, failed, or disputed are just labels on an entry, and those labels are part of the record too.

The task graph can never contain a loop, by design: the function that adds a task, add_task, only accepts links to tasks that already exist, so a loop would need to point to a task that doesn't exist yet — which simply fails with "dependency does not exist." There's no loop-checking code, because there's no way to create a loop in the first place. A related method, snapshot(), which takes a full, timestamped copy of everything, comes back into play in §8.

5What the System Is Entitled to Believe: The Belief System and Its Kernel

Three kinds of believed thing

Not everything the system treats as true carries the same weight, and each kind follows its own rules:

KindDefinitionGoverning rule
Epistemic claimsContestable assertions about the world; what evidence bears on, what can be wrong.Evidence-bearing; climb the belief ladder.
Structural invariantsRules of the system itself: beliefs change only through verdicts; contradictions never vanish silently.Enforced by code; writable by no one.
Contextual anchorsUser-declared fixed facts (§3).Never reworded; labeled user-declared in every output; tensions surfaced, never resolved by the system.

Since each rule applies to a whole category, not to individual cases, no rule needs exceptions: "never reword" applies to every anchor, "the user can't edit this" applies to every invariant, and "debatable, needs evidence" applies only to claims.

The belief ladder

The obvious choice would be a confidence score, since scores are easy to rank and update. But numbers made up by a model don't come out the same way twice, and doing math on them pretends a precision the underlying judgment never had (§2). Xogito trades that precision for consistency: every claim carries one of four labels. Three form a ladder of earned confidence: unverified, supported, and verified. The fourth, contested, isn't a rung on the ladder — it's a flag. Any claim can get this flag at any time, it can't be compared to the other labels, and it just means the claim is caught up in an unresolved conflict.

Which moves are allowed is decided by the kernel, a small block of simple yes/no checks that never talks to an LLM, holds no state, and depends on nothing else. Its rules build in one deliberate imbalance: moving up to supported needs cited evidence; moving up to verified needs two pieces of evidence, or just one if it's a sound logical derivation (a valid derivation is enough on its own, but a single observation never is). There's no shortcut straight from unverified to verified. But dropping a claim's confidence, or flagging it as contested, costs nothing. Doubt is cheap; confidence has to be earned. So when the system has to be wrong, it errs toward too little confidence, which costs some efficiency, rather than too much, which costs correctness.

The kernel only checks whether a move is allowed. The actual change happens through one method on the workspace, update_belief_of_claim — the only door through which any belief can change. It checks the cited evidence exists and is properly linked, asks the kernel if the move is legal, makes the change, and logs it. Every other part of the system can only ask this one method to make a change — nothing else can touch a belief directly.

6Deciding What to Do: Frontier Planning

Planning everything up front, in one call that produces a single ordered task list, fails open-ended problems in two ways. First, quality drops off toward the end of the list: task twelve of fifteen often exists only because the format demanded fifteen entries. Second, the plan is built on ignorance: at planning time, the system doesn't yet know what it will learn from its first investigation, so the back half of any upfront plan is really just a guess.

Both problems come from treating planning as a one-time act, when really the information it needs only arrives gradually. Under frontier planning, each round the planner proposes only the next few tasks that can be done right now and would cut uncertainty the most. A full plan is never made all at once; it takes shape gradually, each round shaped by what the last round just revealed. Making "reduce uncertainty" the standing goal also settles the classic tradeoff between exploring and exploiting, with no tuning needed: the system probes broadly early on, when everything is uncertain, and narrows in on specific gaps later, once most of the picture is clear.

Proposer and judge

Each planning round makes exactly two LLM calls, kept apart by a wall neither can see through.

The proposer gets the approved specification, plus context built by code: one-line summaries of finished work (never the full content), tasks still pending, past rejections with the reasons why, and the current belief table. It's asked to propose the smallest useful set of next tasks — one task is a perfectly good answer, and proposing nothing, with a stated reason, counts as the signal to stop (§12), not as a failure.

The proposer also sees the user's named checklist (§3), with a per-item coverage status computed by code. Each proposed task can claim the item it covers, and code validates that claim — an out-of-range index just means "covers nothing," never a crash. Items that assert an absence ("zero documented evidence") are marked as needing two tasks, worded differently from each other and designed to find the thing claimed absent: one search that mirrors the claim and comes back empty closes nothing, and two searches sharing a wording share blind spots. The proposer may not return an empty frontier while any named item is uncovered. It's also told what workers can actually do — recall documented knowledge and derive conclusions, nothing else — so it never proposes benchmarks or measurements no worker could honestly perform.

Every proposed task must include a done_when condition: something concrete and checkable that marks it done. That condition works two ways: at the start, it screens out vague tasks, since a task whose finish line can't be described shouldn't be proposed at all; later, it's the exact standard the finished work gets checked against (§9). Any task a proposal depends on must appear earlier in the same list, so a dependency loop can't even be written down — and the workspace blocks the one remaining way to sneak one in.

The judge sees only the structured tasks, nothing else, and labels each one sound, vague, duplicate, or invalid_dependency. It only judges whether a task is doable and non-repeated, never whether it sounds promising, and it's told not to reject a task just for being unusual, since odd-but-concrete tasks often catch things a standard approach would miss. Code then checks the judge's own output: every label has to be real, and every task judged exactly once.

Rejected tasks stay on record with their reasons; later rounds see this history and don't propose the same thing again. Anything that depends on a rejected task is rejected too. The judge's label travels with each recorded reason — the checkpoint reads it later to tell "everything left was a duplicate" apart from a genuine stall (§12).

Two safety limits sit above the whole process: a cap per round and an overall budget for the run. Both live in code, are never mentioned in a prompt, and are set at roughly double the highest count normally seen in a healthy run. If either limit is hit, the run stops for a human to check it: a planner proposing thirty tasks at once is malfunctioning, and a malfunction should be visible, not quietly absorbed.

7Dispatch: The Deliberately Thin Scheduler

The scheduler's entire job, in this first version, is simple: ask the workspace which tasks are ready (pending tasks whose dependencies are all finished, a plain code lookup), mark them as in progress, hand them to the executor as a batch, and pass the results along to be integrated. Tasks are dispatched in the order they were added, an order the planner has already made meaningful.

Priority scheduling was considered and set aside for later. Priority only matters when resources are scarce, and this first version runs every ready task each round, so there's nothing to prioritize between yet. When scarcity does become an issue, the useful signal will be a property of the task graph — how many downstream claims and tasks depend on a given task's output — and that data doesn't exist until the evaluator has been running long enough to produce it. The scheduler is deliberately left as a clean spot where parallel work limits, retry rules, and computed priority can be added later without rewiring anything else.

One clear boundary was drawn here: the scheduler owns each task's lifecycle, while the executor owns only the work itself. Executors are the least trusted part of the system, since they sit right next to the LLM, and they're given correspondingly little power to change the system's state.

8Doing the Work: The Executor

Entities and messages

A worker never hands back a changed Task. A Task is an entity — it has an identity and a life story that live inside the workspace. Instead, a worker returns a message: an ExecutionResult that carries what it produced, a one-line summary, the claims that content supports, any missing work it noticed, and, if it failed, the error, how many tries it took, and how long it ran. Messages travel from producer to consumer, then vanish; this one vanishes once it's turned into artifacts, claims, statuses, and a logged entry.

This is what physically enforces the trust boundary. If workers returned changed entities, they'd effectively be holding state themselves. Because they only ever return inert messages, "workers never touch the workspace" isn't a rule someone has to remember — it's built into the system's structure. Verdicts (§9) and evaluation contexts are messages for the same reason.

Parallelize the work, serialize the state

Execution mostly means LLM calls, and those are slow enough that running several at once is worth doing. Looking at the standard options: locking every method would guard writes that already don't conflict by design, at the risk of getting stuck; asyncio makes sense once you're making thousands of calls at once, not just four; and provider batch APIs trade hours of extra waiting for lower cost, but the next planning step needs this batch's results right away. What's left is a thread pool with one rule: do the work in parallel, but make every change to shared state happen one at a time.

At the start of each batch, the workspace is copied once; every worker gets that same frozen copy and never touches the live workspace, so nothing in its code could even accidentally break this rule. Up to four workers run at once; their results are then sorted once by task ID, so integration order, artifact IDs, and the log all come out the same way every run. That consistency, which is what makes two runs comparable when debugging, costs nothing more than one sort.

Results get folded in one at a time, on the main thread: a failed result marks its task as failed; a successful one stores the artifact, adds its claims as unverified (the executor only asserts things — only the evaluator judges them), links the evidence through the paired methods that keep the lists in sync, notes any follow-up work it spotted without adding it to the task graph directly (only the planner can add tasks), and marks the task complete.

Integration also handles agreement. When a new claim says nearly the same thing as an existing one, it doesn't become a twin — the new artifact gets linked to the existing claim as additional evidence. Matching is deliberately cautious: a high bar on word overlap, plus an exact match on negation words, because "asyncio does help" and "asyncio does not help" are one word apart and must never merge. This rule exists because of a live run where two tasks kept agreeing with each other and each agreement became a separate single-source claim — so the two-independent-sources requirement (§9) could never be met by anything, and nothing got established.

Because all writes happen on one thread, there's nothing to lock. In place of locks, there's a tripwire: a check inside the logger, which every write passes through, confirms it's running on the right thread. If a future change accidentally writes from a worker thread, it fails right away, loudly.

Every call runs under a time limit; if it fails, it retries with a longer wait each time; if it keeps failing, it returns a failed result. Failure is just a normal result, not a crash, so one task dying never brings down the whole batch — and the batch itself runs under its own overall time limit. When the provider rate-limits a call and states its own retry delay, that stated delay wins over the local guess: the provider knows its own rate window.

Workers are also told plainly what they can't do: run code, execute benchmarks, measure systems, browse the web, or contact anyone. A task that needs one of those gets "this can't be done from here" as its honest and complete answer — an early run produced a fabricated "benchmark," invented throughput numbers included, which is exactly what this rule and its evaluator-side backstop (§9) exist to stop. Three more output rules came from the same audits: a verdict on a numeric claim states the expected magnitude, never a bare possible/impossible; factual assertions name their sources specifically enough to look up; and any conclusion built on the user's numbers shows its arithmetic.

Tools like web search and code execution haven't been built yet, but their shape is already settled: the LLM call becomes a loop of proposing, checking, and running, so a tool call is just one more kind of proposal for code to check; tool results become artifacts; and model-written code becomes evidence that's stored and can be run again later. The system can gain new abilities without changing who's trusted to do what.

9Judging the Work: The Checkable Evaluator

The regress

The evaluator is an LLM judging another LLM's work — so who checks the evaluator? Adding a second watcher just pushes the same question back one level, forever, and no watcher can turn its own opinion into fact just by declaring it. Xogito takes a third path: every verdict has to end in something code can actually check. The evaluator's opinions never directly change the system's state; only its proposals do, and only after a mechanical check passes them.

A Verdict proposes exactly one change: which claim, what new confidence level, what evidence, what kind of evidence, and why. Two choices matter most: the evidence's ID is assigned by code, not the model, since the evidence is just the artifact under review — this removes a field the model could otherwise make up. And the evidence list can only hold artifact IDs, never other claims, so a verdict can never point to another claim as its support.

That one rule blocks a subtle trap: unverified claim A supports unverified claim B, which later gets cited back in support of A, and confidence inflates out of nowhere through this loop. The rule doesn't need to be checked, because there's no way to even write down a case that breaks it.

Different kinds of support fail in different ways, and that difference changes what "verified" should mean. There are four types in total — really three, plus one that's a flag on another:

Judging quality, independence, and staleness happens within a type — the type decides which rules apply, and quality decides whether those rules are met.

The gauntlet, the dual-pass, and tempo

The evaluator runs once per finished task, looking at an evaluation context: that task's done_when condition, its full artifact, and the linked claims with their current labels. The original problem spec is left out on purpose: an evaluator that knows the overall goal starts judging claims by how useful they are, not by their actual evidence. One call proposes verdicts and, separately, reports whether the done_when condition was met.

Every raw verdict has to pass a fixed set of code-only checks, cheapest first:

  1. Existence. The claim exists; labels are in vocabulary; the cited artifact is actually linked to the claim.
  2. Tempo. At most one rung of promotion per claim per iteration. The kernel forbids skipping a rung; the tempo rule forbids taking two legal steps in one round, so confidence accretes only across iterations.
  3. Type rules. Testimonial verdicts proposing verified are demoted to supported, with the cap recorded. Negative evidence without stated scope is dropped. Quantitative claims cannot reach verified in a single pass. And verified requires evidence originating from at least two different tasks, a code-checkable proxy for framing-independence.
  4. Integrity rules. Added after a live-run audit caught four failure genres slipping through; all apply to promotions only, and all fail toward less confidence. A numeric claim can only rise if its numbers restate the user's own — pinned from the ratified spec by code — or its artifact shows the formula they came from; a number matching neither appeared from nowhere. Claims asserting sign-offs, approvals, or measurements are unpromotable, because no text this system generates is evidence that a human approved something or that a benchmark actually ran. Reaching supported needs evidence from two different tasks — one source echoed five times is still one source — with a lone derivation exempt and an absence claim never exempt. And an artifact produced by a synthesis task is terminal: a conclusion can never circle back as evidence for further claims.

Some checks simply drop a verdict; others knock it down a level and add a note instead. A verdict that overreaches usually still holds real, useful information wrapped in too much confidence, and dialing it back keeps the information while trimming the overreach.

Any verdict that would raise confidence has to clear one more gate, the dual-pass: a second, separate LLM call sees only the claim's text and the evidence itself — never the first pass's reasoning or label — and answers one question: does this evidence actually say or logically require this claim? A second reader shown the first reader's argument tends to just agree with it, so keeping it blind is what makes the check mean something. If the two disagree, the promotion is blocked and the disagreement is logged. Demotions and contested flags skip this second check, in line with how easy it already is to lower confidence.

Verdicts that survive all this get submitted through update_belief_of_claim, where the kernel checks them one more time: the evaluator checks first so one bad verdict doesn't crash the whole batch, and the workspace checks again because it doesn't automatically trust any caller. A cap on how many verdicts can be processed sits above the whole process as a last safeguard.

Bias is handled through required fields, not through instructions. No prompt just tells the model to "be skeptical" — that kind of instruction tends to produce hedging, made-up counter-evidence, and the look of caution rather than the real thing. Instead, workers doing investigation must fill in checkable fields describing what they searched and any counter-evidence they found; done_when conditions are checked to make sure they don't assume an answer (phrased as "the presence or absence of X," never as if X is already true); and needing evidence from two separate tasks for verified means one-sided evidence structurally isn't enough on its own. Neutrality here comes from disciplined handling of information, not from asking a model to watch its own biases.

10Disagreement as a First-Class State

Two claims can each pass every check on their own evidence and still directly contradict each other. The cheap way out, quietly keeping the better-supported claim and dropping the other, is also what models tend to do by default; it's also exactly how a system ends up reporting a conclusion after its strongest counter-evidence was thrown away without a trace. Xogito instead treats the disagreement itself as something worth tracking.

After verdicts are applied, the evaluator scans the entire belief table with one narrow question: which pairs of claims can't both be true? Its prompt warns against the usual false alarms — different emphasis, different scope, and paraphrasing are not contradictions — and it's told that finding nothing is a perfectly fine answer. Code then checks the pairs it finds, and both claims in each genuine pair are moved to contested through that single approved gateway, each carrying a note naming the other, so the whole conflict can be reconstructed later just from the provenance log.

The evaluator only detects contradictions; it never resolves them. A component that both detects and resolves has an incentive to under-detect, since every conflict it finds becomes work it then has to settle — and a judge that found the conflict already arrives with an opinion about it. Resolving it is left to a separate component, the adjudicator.

The adjudicator looks at both sides' evidence trails for a contested pair and decides the outcome, and can create new verify-type tasks if needed — a third kind of task reserved specifically for this purpose, which is exactly why the planner is never allowed to propose one itself. The rule that contradictions are never resolved silently isn't something anyone has to remember to follow; it's simply built into which component is allowed to do what.

11The Judge's Report Card: Calibration

The evaluator will get things wrong in both directions: the dual-pass check will sometimes block a promotion that was actually sound, and the adjudicator will sometimes overturn a promotion that had cleared every other gate. A system that claims to give well-calibrated confidence needs to actually measure how often its labels turn out to be right.

Calibration is a permanent, append-only log kept deliberately separate from the workspace: the workspace holds the state of one problem, while calibration holds the evaluator's track record across many problems over time. Its entries are outcomes — a dual-pass disagreement, an adjudicator confirming or overturning a promotion, a flagged contradiction that was confirmed or dismissed — each tagged with the type of evidence involved, since that's where the patterns tend to show up.

Three uses justify keeping this log:

  • The final report can state, next to its labels, the historical overturn rate for promotions backed by each evidence type.
  • The human tuning the system gets data: testimonial promotions repeatedly dying at adjudication is a specific prompt fix with numbers behind it.
  • A future version can adapt, requiring stricter passes on evidence types with poor records, learning about its own judgment without retraining a weight.

This is really the same idea as provenance, just applied one level up: provenance keeps a record of what happened to the system's state, while calibration keeps a record of what happened to its judgment.

12Ending Well: Checkpoint, Compression, Report

Checkpoint

The checkpoint decides whether to continue, based on signals that are already available: an empty set of next tasks with a stated reason (either the success criteria are met, or the remaining uncertainty just can't be reduced further); a stall, meaning every proposal in a round got rejected, visible as a batch of rejected tasks with reasons; a buildup of done_when failures; or simply hitting the hard limit on rounds, a fixed constant in the code.

Two refinements came from watching real runs end. A round where every proposal was rejected specifically as a duplicate of covered ground counts as a finished frontier, not a stall — there was nothing new left to propose, and the judge's labels on the rejections are what let code tell the difference. And the run may not stop as a success while any item on the user's named checklist (§3) still lacks a completed covering task — with absence-type items needing two. An empty frontier that skipped a named question isn't done; it's incomplete, and the report says so, naming the dropped items.

The checkpoint is also the one approved place where success criteria can be revised: sometimes, doing the actual work reveals that the approved criteria were subtly off, and the real question turns out to be slightly different from the one originally asked. The system can propose a revision, but only the user can approve it, through the same process used for the original specification. Letting a system quietly redefine its own goals without a human signing off is exactly how optimizing systems drift off course. Making edits mid-run to tasks or claims was set aside for later, but its design is already settled: such changes will be logged as events rather than silent edits, will go through the same checks as everything else, and the size of a change's impact will be calculated and shown before it's applied.

Compression

Context windows run out, but the reasoning process shouldn't have to stop because of it. The compressor follows one rule: structured state carries over between rounds untouched; only the narrative gets shortened. The belief table, contested pairs, and tension flags are carried forward exactly as they are, since a summarized belief table would just be a different, worse belief table. Free-form context, like what's been tried and what the situation looks like so far, can be safely shortened because none of it carries a label the system depends on. Compression has to lose nothing on the dimensions that actually matter, and those dimensions are a short, known list.

Report

The report displays the reasoning itself; it doesn't just summarize it. Claims appear with the confidence labels they actually earned and the evidence chains behind them, all traceable through provenance. Anchors are shown labeled as user-declared assumptions, with any tension flags displayed rather than smoothed over. Findings of "nothing was found" state exactly what was searched. Claims involving numbers are phrased as what they really are, such as "two independent sources assert this," never dressed up as a statistical conclusion the system never actually computed. The calibration history gives context for how much to trust each label. Contested claims that were never resolved are shown as contested, with both sides laid out, because an honestly documented disagreement is more useful than a made-up resolution presented as a settled conclusion.

The answer at the top of the report is written from the established claims only — and the writing call is shown the user's named questions and declared facts, verbatim. It has to address every named item in order, saying plainly when the findings didn't settle one rather than guessing or reversing what a finding states, and it shows its arithmetic in the user's own numbers. This rule exists because an early synthesis, shown only the goal and the findings, paraphrased a named question from memory and produced a verdict backwards from what the run's own tasks had concluded.

Even a run that gets halted, whether by a tripped safety limit or a crash, still produces its report. An audit trail that vanishes exactly when something goes wrong would be worthless.
