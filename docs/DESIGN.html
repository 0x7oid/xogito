Xogito: The Architecture of Auditable Reasoning

Design decisions, alternatives, and trade-offs.

Part I · The problem

1Reasoning Fails Through Bookkeeping

Ask a frontier model a hard question and it will often produce a chain of reasoning in which every individual step is competent: correct arithmetic, plausible facts, locally valid logic. The conclusion is still wrong, often confidently wrong, and nothing in the output indicates which premises deserve trust.

Inspected across many failures, the pattern is consistent. The model does not fail at individual steps; it loses track of its own epistemic state. It treats a guess made early as an established fact later. It resolves contradictions between its own statements by silently dropping one. It cites its own speculation as support for further speculation. At the end, it cannot report which conclusions rest on evidence and which on momentum.

These are bookkeeping failures: failures of the machinery that should track what is believed, on what basis, with what confidence, and what conflicts with what. Xogito is built on one premise: if that bookkeeping is made external, explicit, and mechanically enforced, ordinary LLM capability is sufficient for reliable reasoning.

The system's output is a set of auditable conclusions. Every claim in the final report carries a confidence label earned through gated transitions, backed by evidence that can be traced, produced by a process whose every state change was recorded. The design question throughout is therefore "what architecture makes it impossible for the bookkeeping to silently fail?", not "how do we make the model smarter?"

2The First Commitment: LLM Proposes, Code Disposes

Should model output be allowed to mutate system state directly? Most agent frameworks answer yes, with validation added where problems appear. That approach works when the cost of a bad write is low. Here a bad write is a corrupted belief, and corrupted beliefs are the product failing. Agents passing prose to each other while updating shared memory make the failure worse: errors launder through repetition, and after a few hops the origin of a false belief is unrecoverable.

Xogito applies, at every boundary, the separation used in secure systems: the entity that proposes an action never authorizes it. Model outputs are always structured proposals (candidate tasks, candidate verdicts, candidate formalizations), and every proposal passes deterministic Python validation before it can touch state. The model holds all of the judgment and none of the authority; the code holds all of the authority and exercises no judgment.

Three corollaries recur throughout the design:

Make illegal states unrepresentable.
Where possible, do not check that a rule is followed; construct the system so the violation cannot be expressed. A dependency cycle that cannot be written beats a cycle detector. Structure has no failure rate. Vigilance does.
Labels, never numbers.
Model judgments are expressed in small closed vocabularies (fits / partially fits / doesn't fit; strong / adequate / weak), never scores. Numeric confidence composes well on paper, but a model-produced "7/10" drifts under rewording, fails to reproduce across runs, and invites downstream arithmetic (averaging, weighting, thresholding) that the underlying judgment never licensed.
Constants belong to code.
Every cap, threshold, and budget lives in Python constants (parametres.py) and never appears in a prompt. A model told "at most ten tasks" treats ten as a target and pads to meet it. A model bounded by an invisible fuse proposes what the problem needs. Graders' rubrics are never shown to generators; this is the standing defense against Goodhart's law.

The pipeline these commitments produce: a vague question becomes a ratified specification (§3); a plan, execute, evaluate cycle accumulates evidence-backed claims (§6 through §10); the run ends in a report that exposes its own reasoning (§12).

Part II · The machine

3From Words to Problem: Intake and Formalization

Formalization is underdetermined

A request like "help me decide whether to launch this product in Algeria" must become a goal, constraints, and success criteria before any planning can occur. The direct implementation is one LLM call: "restate this as a formal problem." For well-posed inputs that is adequate. It fails on the inputs that matter, because there is no such thing as the formalization of an informal problem. The same goal can be rendered as an optimization (find the best strategy), a constraint-satisfaction problem (find any strategy clearing these bars), a game (competitors respond), a control loop (an ongoing process), or a satisficing search (find good enough, stop). These framings define different success conditions, generate different tasks, and produce different kinds of answers.

Carnap called this situation explication: replacing a vague concept with a precise one, where candidates are not correct or incorrect, only better or worse along four criteria (similarity, exactness, fruitfulness, simplicity). A single-call formalizer collapses this underdetermination silently. It picks a framing, usually optimization because that genre dominates training data, and the user never learns that a contestable choice was made. The largest ambiguity in the pipeline would be resolved by accident.

The five-phase pipeline

Phase 1: Characterize. One LLM call extracts structural characteristics: does success admit degrees or is it pass/fail; one actor or several with conflicting interests; one-shot decision or feedback process. The call is catalog-blind. It has no knowledge of which formal structures exist, because a model that knows optimization is on the menu will extract characteristics that justify optimization. Outputs are enums, and code validates each against the allowed vocabulary, since Phase 2 compares exact strings.

Phase 2: Filter. No LLM. A static, code-owned catalog of structures (optimization, CSP, satisficing, game theory, control loop) carries applicability predicates, plain Python functions over the characteristics. Filtering is set intersection. The general lesson: once a question has been reduced to structured data, push it out of the model. Every question answered by code is one the model cannot answer inconsistently.

Phase 3: Classify and select. One batched call labels each surviving structure fits / partially fits / doesn't fit, and is explicitly told that labeling everything "doesn't fit" is acceptable, which counters the model's tendency to force matches. Code buckets by label tier and truncates to a cap the model never saw. If the cut falls between two structures with the same label, the record notes boundary tension: the truncation was arbitrary at the margin, and the record says so.

Phase 4: Generate. Each selected structure receives its formalization, instructed to treat structures in isolation and to report what the framing drops or alters. A formalization that admits what it discarded is auditable; one that claims perfect fit is hiding its losses.

Phase 5: Evaluate and decide. A final call labels each candidate on Carnap's four criteria (strong / adequate / weak) and is forbidden from declaring a winner; code counts labels and picks. If the top two candidates tie on their label profile, the result is contested and the decision block reads selected_structure_id: None. The system does not break ties.

Contested records and ratification

A single sanctioned accessor, get_final_formalization, is the only way downstream code can extract the chosen formalization, and it raises on a contested record. No caller can proceed on a tie because proceeding is unimplementable. Resolution is human: the user is shown the tied framings and their consequences ("finding the single best option" versus "finding any option that clears these bars") and picks one.

Whether or not the record was contested, the completed problem specification is then presented for ratification: the user confirms the extracted goal, constraints, success criteria, and assumptions. The specification is stamped ratified: true, and a pure predicate, assert_spec_ratified, checks that stamp at the entry of every downstream component. The check is redundant by design. Forgetting it anywhere would fail silently, so it appears everywhere.

Contextual anchors

Facts the user declares fixed ("the budget is $40k") are anchors. They are split from raw text by code on delimiters, never by an LLM that might normalize them, then stored verbatim and injected verbatim into every prompt that uses them, labeled as user-declared. The model may never reword an anchor: rewording is reinterpreting, which is precisely the silent drift the system exists to prevent. When later evidence conflicts with an anchor, the system raises a tension flag; it does not overrule the user.

4Memory That Cannot Lie: The Workspace

Two questions shape everything downstream: where does reasoning state live, and what needs to persist? The default in most multi-agent systems is per-agent state, merged when agents hand off. That works when outputs are independent and combined once. Here they are not independent. When two executors reach conflicting conclusions about the same fact, some module must learn of the disagreement, the final report needs one authoritative version to cite, and a challenged conclusion must be traceable to its origin. Distributed state with no single source of truth produces beliefs nobody decided to hold.

The workspace is a single store holding the entire epistemic state of a run, mutated only through a narrow set of guarded methods, with every mutation logged. It resembles the global-workspace pattern from cognitive science (specialists reading and writing one shared blackboard), with one hardening: agents do not write. Agents submit, and the workspace's own methods decide what is written.

Entities

The workspace stores three entity kinds, each a plain dataclass:

  • Task: an assignment. Description, kind, dependencies, lifecycle status, and a checkable completion contract (done_when, supplied by the planner, §6).
  • Claim: a tracked assertion. Statement, belief label, and the list of artifacts serving as evidence.
  • Artifact: a work product. Full content, a one-line summary, and the producing task.

The claim-artifact link is bidirectional: a claim lists its evidence artifacts, and an artifact lists the claims it supports. Both lists are written by exactly one method pair, link_evidence and unlink_evidence. Two synced lists maintained by a single writer cannot desynchronize; the same lists maintained by any caller who must remember to update both eventually will, silently. A junction-table design, with explicit link objects, would support per-link metadata and is the standard relational answer; it was deferred until links actually need metadata, a threshold the evidence-type system (§9) is expected to cross.

Access rules

Every getter returns deep copies, so external code can never hold a live reference to workspace internals. External mutation is not forbidden; it is ineffective, which is stronger. Writers use the internal dictionaries directly, because a writer that read through its own copying getter would mutate a copy and discard it (the "copy trap"). Every write method validates referential integrity before mutating (a claim citing a nonexistent artifact is rejected) and calls the provenance logger itself, without trusting the caller to remember.

Provenance is the append-only log of every state change: which actor kind did what, to which entity, when. It makes "where did this belief come from?" a query rather than an investigation. Nothing is ever deleted from the workspace: not failed tasks, not rejected proposals, not retracted claims. Deletion destroys history. Rejection, failure, and contestation are statuses, and statuses are provenance.

The task graph is a DAG by construction. Tasks are inserted in order, and add_task accepts only dependency identifiers that already exist. A cycle would require a forward reference, and a forward reference fails as "dependency does not exist," so no cycle detector exists, because no cycle can be written.

The workspace also provides snapshot(), a timestamped deep copy of everything; its role appears with concurrency in §8.

5What the System Is Entitled to Believe: The Belief System and Its Kernel

Three kinds of believed thing

Not everything the system holds true has the same standing, and each kind obeys different rules:

KindDefinitionGoverning rule
Epistemic claimsContestable assertions about the world; what evidence bears on, what can be wrong.Evidence-bearing; climb the belief ladder.
Structural invariantsRules of the system itself: beliefs change only through verdicts; contradictions never vanish silently.Enforced by code; writable by no one.
Contextual anchorsUser-declared fixed facts (§3).Never reworded; labeled user-declared in every output; tensions surfaced, never resolved by the system.

Because rules quantify over kinds, no rule needs exception clauses: "never reword" applies to anchors, "never user-writable" to invariants, "contestable and evidence-bearing" to claims only.

The belief ladder

How should uncertainty be represented? A continuous confidence score is the common choice and supports ranking and Bayesian updating; it was ruled out here for the reasons in §2 — model-produced numbers do not reproduce, and arithmetic over them implies precision the judgments lack. Xogito trades resolution for reproducibility: every claim carries one of four labels. Three form a ladder of earned confidence: unverified, supported, verified. The fourth, contested, is a flag state, not a rung: reachable from anywhere, comparable to nothing, meaning the claim is party to an unresolved contradiction.

Legal transitions live in the kernel, a small module of pure predicates with no LLM contact, no state, and no dependencies. Its transition table encodes one asymmetry. Promotion to supported requires cited evidence; promotion to verified requires two pieces of evidence, or one if deductive (a valid derivation suffices alone; an observation never does); there is no path from unverified directly to verified. Demotion and contestation require nothing. A system where doubt is cheaper than confidence fails toward under-confidence, which costs efficiency, rather than over-confidence, which costs correctness.

The kernel checks transitions; the workspace's update_belief_of_claim, the single method through which any belief changes, verifies the cited evidence exists and is linked, invokes the kernel, mutates, and logs. Everything else in the system can only submit proposals to that method.

6Deciding What to Do: Frontier Planning

Complete up-front planning, one call producing an ordered task list, is reasonable when the task structure is known before execution begins. Run it on open-ended problems and two failures appear. Quality decays down the list: task twelve of fifteen is filler generated because the format demanded fifteen entries. And the plan is built from ignorance: at planning time the system knows nothing it will know after the first investigation, so the back half of any up-front plan is speculation.

Both failures come from treating planning as an act when the information arrives as a process. Under frontier planning, each iteration the planner proposes only the next few tasks that are workable now and would most reduce uncertainty. The full plan is never generated; it emerges across iterations, each informed by what execution just learned. This also resolves the exploration/exploitation trade-off without a tuning parameter: uncertainty-reduction as the standing objective yields broad probes early, when everything is uncertain, and targeted work late, when only specific gaps remain.

Proposer and judge

Each planning iteration makes exactly two LLM calls, separated by an information wall.

The proposer receives the ratified specification and a code-built context: one-line summaries of completed artifacts (the planner never reads full content; the summary field exists for this), pending tasks, previously rejected proposals with reasons, and the belief table. It is asked for the minimum workable frontier and told that one task is a fine answer and zero is legitimate. An empty frontier with a stated reason is the loop's stopping signal (§12), not a failure.

Each proposed task must carry a done_when: a concrete, checkable completion condition. The field serves twice. At generation it filters vague tasks, since a task whose completion cannot be stated should not be proposed. At evaluation it is the contract the work is inspected against (§9). Dependencies are expressed as indices into the proposal list itself, restricted to earlier positions, making cycles unwritable one layer before the workspace makes them uninsertable.

The judge reviews only the structured tasks and labels each sound, vague, duplicate, or invalid_dependency. Its prompt states: do not reject tasks for being unusual, indirect, or seemingly unpromising; judge executability and redundancy, not promise. Without that restriction, judged planning converges on conventional tasks, and unconventional-but-concrete probes are the ones that find what standard analyses miss. Code then validates the judge's own output: every label in vocabulary, every index judged exactly once.

Rejected proposals are inserted as tasks with status rejected and the reason attached; future planner iterations see them and stop re-proposing them. A task depending on a rejected task is rejected transitively, with that recorded as the reason.

Two fuses sit above the process: a per-iteration proposal cap and a global task budget, both code-owned, both invisible to prompts, both set at roughly double the healthy observed maximum. A tripped fuse raises an exception that halts the run for human review. A planner proposing thirty tasks is malfunctioning, and malfunctions are surfaced, not absorbed. Every loop component carries a fuse with these semantics.

7Dispatch: The Deliberately Thin Scheduler

The scheduler's entire v1 job: query the workspace for ready tasks (pending tasks whose dependencies are all completed, a pure code query), mark them in-progress, hand them to the executor as a batch, and route results into integration. Dispatch order is FIFO on insertion order, which the planner already made meaningful.

Priority scheduling was considered and deferred. Priority matters only under scarcity, when not everything ready can run, and the v1 loop runs every ready task each iteration, so there is nothing to arbitrate. When scarcity arrives, the useful priority signal is a computed graph property (fan-out: how many downstream claims and tasks depend on a task's output), and that data does not exist until the evaluator has run for a while. Building priority now would mean building it on guesses. The scheduler remains a named seam where parallel budgets, retry policy, and computed priority can land without rewiring.

One boundary decision: the scheduler owns task lifecycle; the executor owns only the work. Executors are the least trusted components, being LLM-adjacent, and hold correspondingly little state-mutation authority. The executor produces, the scheduler bookkeeps, the workspace guards.

8Doing the Work: The Executor

Entities and messages

A worker does not return a mutated Task. A Task is an entity: it has identity and a lifecycle inside the workspace. A worker returns a message: an ExecutionResult carrying the content produced, a one-line summary, the factual claims the work asserts, any missing work the worker noticed, and, on failure, the error, attempt count, and elapsed time. Messages travel from producer to consumer and dissolve; this one dissolves at integration into artifacts, claims, statuses, and provenance.

The distinction enforces the trust architecture physically. If workers returned mutated entities, workers would hold and modify state. Because they return inert messages, "workers never touch the workspace" is the type signature, not a convention. Verdicts (§9) and evaluation contexts are messages for the same reason.

Parallelize the work, serialize the state

Task execution means LLM calls: slow I/O, worth parallelizing, and the usual source of concurrency bugs. Three standard designs were weighed. The monitor pattern, a mutex on every workspace method, is correct and widely used; here it would guard writes that the design below already makes non-concurrent, at the price of deadlock risk and masked ordering bugs. asyncio is the right tool at thousands of concurrent calls; adopting it for four means rewriting most of the codebase for no throughput gain. Provider batch APIs trade latency for cost with turnaround measured in hours; the loop's next planning step consumes this batch's outputs, so that latency is disqualifying. What remains is a thread pool with one rule: parallelize the work, serialize the state.

At batch start the workspace is snapshotted once; every worker receives that same frozen view plus its task, and builds its own execution context. Workers hold no reference to the live workspace. There is no variable in a worker's scope through which a violation could be typed. Up to four workers run in a thread pool; results are collected as they finish and sorted once by task identifier, so integration order (and therefore artifact IDs and provenance order) is deterministic across runs regardless of network timing. Reproducibility is what makes two runs comparable when debugging, and it costs one sort().

Integration runs serially on the main thread. For each result in ID order: a failed result marks its task failed with the error recorded. A successful one stores the artifact, enters each asserted claim as unverified (the executor asserts; only the evaluator judges), links claim to artifact through the synced-list method, surfaces proposed follow-up work without inserting it (the planner is the sole entry to the task graph; executor suggestions are recorded, not enacted), and then marks the task complete.

Because all writes occur on one thread, there is nothing to lock. The workspace carries a tripwire in place of locks: an owner-thread assertion inside the provenance logger, through which every write passes. If a future refactor writes from a worker thread, the failure is immediate and loud, not a silent race.

Failure handling: every LLM call runs under a timeout, since a hung request would otherwise occupy a worker slot indefinitely. Failures retry with exponential backoff. A worker that exhausts retries returns a failed result. Failure is a result, not an exception, so one task's death never kills the batch. The whole batch runs under a wall-clock fuse.

Tools (web search, code execution) are deferred with the shape decided: the single LLM call becomes a propose-validate-execute loop, tool calls being one more proposal category for code to dispose of. Tool results become artifacts, and model-generated code becomes stored, re-runnable evidence. Capability grows without changing the trust architecture.

9Judging the Work: The Checkable Evaluator

The regress

The evaluator reads artifacts and proposes belief transitions. But the evaluator is an LLM, the same fallible kind of judgment it is judging. Who evaluates the evaluator? A second watcher regresses infinitely; no watcher makes the evaluator's opinions facts by fiat. The framework takes a third position: every verdict must terminate in something code can check. The evaluator's opinions never become state; its proposals do, after mechanical scrutiny.

A Verdict, the evaluator's message type, proposes one transition: claim, target belief, evidence, evidence type, rationale. Two structural decisions in this dataclass carry most of the weight. First, evidence identifiers are assigned by code, not requested from the model: the evidence for a verdict is the artifact under evaluation, which removes a hallucinatable field. Second, the evidence list holds artifact IDs only, so a verdict cannot cite another claim as evidence.

The type constraint eliminates bootstrapping: unverified claim A supporting unverified claim B which later supports A, confidence inflating through self-citation. The rule needs no checking because its violation cannot be written.

The gauntlet, the dual-pass, and tempo

The evaluator runs per completed task on an evaluation context: the task's done_when, its artifact in full, and the linked claims with current labels. The problem specification is deliberately absent, because an evaluator that knows the goal starts judging claims by their usefulness to the goal, not by their evidence. One call proposes verdicts for the linked claims and separately reports whether the artifact satisfied its done_when contract.

Every raw verdict passes an ordered sequence of pure-code checks, cheap tests first:

  1. Existence. The claim exists; labels are in vocabulary; the cited artifact is actually linked to the claim.
  2. Tempo. At most one rung of promotion per claim per iteration. The kernel forbids skipping a rung; the tempo rule forbids taking two legal steps in one round, so confidence accretes only across iterations.
  3. Type rules. Testimonial verdicts proposing verified are demoted to supported, with the cap recorded in the rationale. Negative evidence without stated scope is dropped. Quantitative claims (heuristic: digits in the statement) cannot reach verified in a single pass. verified for any claim requires evidence originating from at least two different tasks, a code-checkable proxy for framing-independence.

Some checks drop a verdict; others demote and annotate. A too-ambitious verdict usually contains real information wearing too much confidence. Demotion keeps the information and sheds the overreach.

Verdicts that would raise a belief face one more gate, the dual-pass: a second LLM call receives only the claim text and the evidence content, not the first pass's rationale or proposed label, and answers one question: does this evidence state or entail this claim? A second reader shown the first reader's argument becomes an agreement machine, so blindness is the mechanism. Disagreement kills the promotion and is logged. Demotions and contestations skip the dual-pass, consistent with the ladder's asymmetry.

Surviving verdicts are submitted through update_belief_of_claim, where the kernel checks legality again. The evaluator pre-checks so one bad verdict is dropped without crashing the batch; the workspace re-checks because it trusts no caller. A verdict-count fuse sits above the whole process.

Bias is handled with fields, not instructions. No prompt says "be skeptical" or "watch for confirmation bias"; such exhortations produce hedging, invented counter-evidence, and vigilance theater. Workers on investigation tasks must fill checkable fields for search scope and counter-evidence encountered; done_when conditions are validated as outcome-neutral (establishing "the presence or absence of X," never presupposing); and the two-task requirement for verified makes one-sided evidence structurally insufficient. Neutrality comes from information hygiene and code-inspectable evidence properties, not from asking a model to police its own psychology.

10Disagreement as a First-Class State

Two claims can each survive the gauntlet on their own evidence and still contradict each other. What should happen when independent reasoning paths disagree? Silent resolution (keep the better-supported claim, drop the other) is the cheapest option and the one models perform by default; it is also how a system comes to report a conclusion whose strongest counter-evidence was discarded without record. Xogito makes the disagreement itself a tracked state.

After verdicts are applied, the evaluator makes one further pass over the whole belief table with a single narrow question: which pairs of claims cannot both be true? The prompt warns against the common false positive (different emphasis, different scope, and paraphrase are not contradictions) and states that an empty answer is acceptable. Code validates the pairs (both claims exist; no claim contradicts itself), and both members of each genuine pair move to contested through the single door, each carrying a rationale naming its opponent, so the conflict is reconstructable from provenance alone.

The evaluator detects; it never resolves. A detector that also judges has an incentive to under-detect, since every conflict it finds is work it must then settle, and a judge who found the conflict arrives with a formed opinion. Resolution belongs to a separate component, the adjudicator.

The adjudicator examines a contested pair's evidence trails and decides the outcome, optionally spawning verify-kind tasks: the third task kind, reserved for this trigger, which is why the planner is forbidden from proposing one. Verification work is born from contradiction, not from planning. The structural invariant "contradictions are never silently resolved" is thus implemented as a division of capability, not as a rule anyone must remember.

11The Judge's Report Card: Calibration

The evaluator will be wrong in both directions. The dual-pass will block sound promotions, and the adjudicator will overturn promotions that survived every gate. A system that promises calibrated confidence must measure how often its labels turn out to have been right, not merely assert that they mean something.

Calibration is an append-only log, deliberately outside the workspace: the workspace is one problem's state, while calibration is the evaluator's track record across problems. Entries are outcomes (a dual-pass disagreement; an adjudicator overturning or confirming a promotion; a flagged contradiction dismissed as spurious or confirmed as real), each tagged with the evidence type involved, because types are where patterns concentrate.

Three consumers justify the log:

  • The final report can state, next to its labels, the historical overturn rate for promotions backed by each evidence type. Measured confidence, not asserted confidence.
  • The human tuning the system gets data. If testimonial promotions keep dying at adjudication, that is a specific prompt fix with numbers behind it.
  • A future version can adapt, requiring stricter passes on evidence types with poor records, learning about its own judgment without retraining a weight.

The mechanism is the provenance pattern one level up: provenance records what happened to state, calibration records what happened to judgment.

12Ending Well: Checkpoint, Compression, Report

Checkpoint

The checkpoint decides continuation from signals already flowing: an empty frontier with a stated reason (the planner judges the success criteria satisfied or the remaining uncertainty irreducible); a stall (every proposal rejected, visible as a batch of rejected tasks with reasons); accumulating done_when failures; or the hard iteration ceiling, a code-owned constant.

The checkpoint is also the sanctioned venue for criteria revision. Execution sometimes reveals that the ratified success criteria were subtly wrong, the real question adjacent to the asked one. The system may propose a revision; only the user may ratify it, through the same ceremony as the original specification. Self-modifying goals without human sign-off is how optimizers drift, so the door exists and carries the same lock as every other door. Mid-run human edits to tasks or claims were analyzed and deferred with the design settled: revisions will be events, not edits, entering through the same gates as everything else, with the blast radius of a change computed and displayed before it is applied.

Compression

Context windows end; iterations shouldn't have to. The compressor's rule is one distinction: structured state crosses iteration boundaries verbatim; only narrative is compressed. The belief table, contested pairs, and tension flags are carried untouched. A summary of a belief table is a different and worse belief table: "mostly supported claims about the market" deletes exactly the labels the system exists to maintain. Prose context (what has been tried, what the terrain looks like) compresses safely because it carries no load-bearing labels. Compression must be lossless on the dimensions that matter, and those dimensions are enumerable.

Report

The report exhibits the reasoning; it does not summarize it. Claims appear with their earned labels and evidence chains, traceable through provenance. Anchors appear labeled as user-declared premises, with tension flags shown, not smoothed over. Negative findings state their search scope. Quantitative claims are phrased as what they are ("two independent sources assert this," never a statistical conclusion the system did not compute). Calibration history contextualizes the labels. Contested claims that were never resolved appear as contested with both sides shown, because a documented conflict is more useful than an arbitrary resolution presented as a conclusion.

A halted run (fuse trip, crash) still produces its report. An audit trail that disappears exactly when things go wrong would be worthless.

Part III · The whole

13The Complete Architecture

The workspace and kernel sit at the center; every belief change passes their single door (§4, §5). The loop's agents surround them, each holding judgment without authority: the planner proposes tasks (§6), the scheduler dispatches them (§7), the executor produces messages from a frozen snapshot (§8), the evaluator proposes verdicts through the gauntlet and dual-pass (§9), and the adjudicator settles the conflicts the evaluator can only flag (§10). Intake and formalization gate the loop's entrance with a ratified specification (§3). Calibration sits outside the workspace because it outlives any single run (§11). The checkpoint, compressor, and report end the run (§12). The orchestrator sequences these components and passes data; it makes no judgments and contains no LLM calls.

14Cross-Cutting Principles

Several rules appear in every component and belong to none of them.

Authority is proportional to checkability. The kernel, pure code, holds absolute authority and makes no judgments. Models make all the judgments and hold no authority. Workspace methods, gauntlets, and validators are the machinery in between. Trust is never extended to a component, only to a check.

Structure over vigilance. Cycles are unwritable, not detected (§4, §6); claim-citation is unrepresentable, not forbidden (§9); workers are physically unable to reach state, not instructed to abstain (§8); illegal transitions are rejected by a table, not discouraged by a prompt (§5). Every rule that can be moved from "remembered" to "impossible to violate" is moved.

Information hygiene. The characterizer never sees the catalog (§3); the proposer never sees the judge's rubric and the judge never sees the proposer's reasoning (§6); the dual-pass never sees the first pass (§9); the evaluator never sees the goal (§9); generators never see caps (§2). What a component is not shown is as much a design decision as what it is shown.

States, not deletions; fuses, not degradation; messages, not shared objects. History is preserved because audit is the product (§4). Abnormal volume halts for a human because a system that degrades gracefully into nonsense reports the nonsense confidently (§6, §8, §9). Components communicate through dissolving messages because shared mutable objects erode trust boundaries (§8, §9).

Defer from evidence, not imagination. Tool registries (§8), priority scheduling (§7), junction tables (§4), adaptive calibration (§11), and mid-run edits (§12) were each designed to the point of knowing their shape, then not built, awaiting a concrete second instance. Features grow from observed need.

Every component above (the ladder, the kernel, the gauntlet, the provenance log, the contested state, the calibration file) is a piece of bookkeeping made external, explicit, and mechanically enforced. None of it makes the model smarter; all of it constrains where the model's fallibility can propagate.
